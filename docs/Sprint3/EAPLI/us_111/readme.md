# US111 — Generate a Simulation Report

## 1. Context

This user story is assigned to **Sprint 3** as part of the EAPLI-related work. It is the first time this feature is being developed. The objective is to allow a Flight Control Operator to receive a summary report of the simulation results, so that they can determine if the programmed flights are safe to run.

**Assigned to:** Flight Control Operator feature set

## 2. Requirements

**US111** As a Flight Control Operator, I want to receive a summary of the simulation results so that I can determine if the programmed flights are safe to run.

### Acceptance Criteria

- **US111.1** The system must generate a report and store it in a file.
- **US111.2** The report must include the total number of flights and their execution status.
- **US111.3** If safety violations occur, the report must list the timestamps and positions of each violation.
- **US111.4** The report must indicate whether the scheduled flights plan passed or failed validation.

### Dependencies/References

- **US100** — Simulate flights in a given area (simulation must exist before a report can be generated).
- **US101** — Capture and process flight movements (position data used in the report).
- **US102** — Detect aircraft safety violations in real time (violation events included in the report).
- **US109** — Generate and store final simulation report (US111 is the EAPLI-side trigger; US109 is the SCOMP-side implementation of the report generation thread).
- **US030** — Authentication and authorization infrastructure.
- **NFR08** — Database by configuration (in-memory vs RDBMS).
- **NFR09** — Authentication and authorization must be enforced.

## 3. Analysis

### 3.0 LLM Assistance

Generative AI (Claude, Anthropic) was used to support the analysis and design of this user story.

**Prompt 1:** "How should a simulation report be modelled in a DDD context? Should it be an aggregate or a value object?"

**LLM suggestions adopted:**
- `SimulationReport` is modelled as an entity (aggregate root) since it has identity, is persisted to a file, and is queried independently.
- The report is generated by reading aggregated simulation data after the simulation concludes, not during it.

**Decisions made by the team:**
- The report is generated on demand by the Flight Control Operator after the simulation finishes, triggered via the EAPLI UI.
- The actual data aggregation (flight statuses, violation events) is produced by the SCOMP component (US109 report generation thread) and made available to the EAPLI layer via a shared data structure or file.
- The EAPLI layer is responsible for presenting the report to the user and confirming the file was saved.

### 3.1 Domain Rules

| Rule | Where enforced |
|------|---------------|
| Report can only be generated after simulation concludes | `GenerateSimulationReportController` — checks simulation status |
| Report must include total number of flights and their statuses | `SimulationReport` — aggregates `FlightExecutionStatus` list |
| Safety violations must include timestamp and position | `SafetyViolationEvent` value object |
| Overall pass/fail result must be clearly indicated | `SimulationReport.validationResult()` |
| Report must be saved to a file | `SimulationReportFileWriter` |

### 3.2 Key Domain Concepts

- **SimulationReport** — Aggregate root. Contains the total flight count, a list of `FlightExecutionStatus` entries, a list of `SafetyViolationEvent` entries, and an overall `ValidationResult` (PASS/FAIL).
- **FlightExecutionStatus** — Value object. Represents the execution outcome of a single flight (flight id + status: COMPLETED, ABORTED, VIOLATION).
- **SafetyViolationEvent** — Value object. Represents a detected safety violation, including timestamp, positions of involved aircraft, and involved flight ids.
- **ValidationResult** — Enum. Either `PASS` (no violations) or `FAIL` (one or more violations detected).
- **Simulation** — The simulation context that the report is generated for, identified by a `SimulationId`.

### 4.2 Acceptance Tests

**AT1 — Report generated successfully with no violations (US111.1, US111.2, US111.4)**

Given a concluded simulation with 5 flights, all completed without violations,
When the FCO triggers report generation,
Then the system generates a report with total flights = 5, all statuses COMPLETED, validation result = PASS, and saves it to a file.

**AT2 — Report includes safety violations (US111.3)**

Given a concluded simulation where 2 safety violations were detected,
When the FCO triggers report generation,
Then the report lists both violation events with their timestamps and positions, and validation result = FAIL.

**AT3 — Report file is saved (US111.1)**

Given a successfully generated report,
When the report generation concludes,
Then a file containing the report is stored in the configured output directory, and the UI confirms the file path to the user.

**AT4 — Unauthorized access is rejected (US030)**

Given a user not authenticated as FLIGHT_CONTROL_OPERATOR,
When they attempt to generate a simulation report,
Then the system denies access with an authorization error.

## 6. Integration/Demonstration

1. Run a simulation (US100) until it concludes.
2. Log in as Flight Control Operator.
3. Select "Generate Simulation Report" from the menu.
4. Select the concluded simulation.
5. System generates and saves the report → file path shown to the user.
6. Open the file and verify: total flights, statuses, violation events (if any), PASS/FAIL result.

## 7. Observations

- US111 (EAPLI) and US109 (SCOMP) are closely related: US109 describes the report generation thread in C (SCOMP side), while US111 describes the Java/EAPLI side trigger and presentation. The integration point is the shared data produced by the SCOMP simulation process and consumed by the EAPLI controller.
- The report file format (e.g., plain text, JSON) should be agreed with the team to ensure consistency between the SCOMP output and the EAPLI presentation.
- Per NFR08, both `JpaSimulationReportRepository` and `InMemorySimulationReportRepository` must be provided.
